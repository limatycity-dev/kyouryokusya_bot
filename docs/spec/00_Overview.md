# 00_overview — 文明BOTの全体像

## 1. 文明BOTの思想（Philosophy）

文明BOTは、Discord コミュニティにおける  
**「努力の可視化」「文化の蓄積」「行動の継続性」**  
を実現するための **文化装置** である。

Discord は本来、会話が流れ、努力が埋もれやすい構造を持つ。  
文明BOTはその弱点を補い、

- 行動が記録される  
- 努力が文化として残る  
- メンバーの挑戦が文明の成長として可視化される  

という **“文明としての Discord コミュニティ”** を成立させる。

文明BOTはタスク管理ツールではない。  
**コミュニティの文化を育てるための OS（Operating System）** である。

---

## 2. 文明BOTのアーキテクチャ（System Architecture）

文明BOTは、以下の 5 つの Feature を中心に構成される。

- **01_setup**：文明の初期化・設定  
- **02_quests**：挑戦（タスク）の管理  
- **03_ranking**：努力の可視化（ランキング・週次レポート）  
- **04_interview**：メンバーの内面を引き出す仕組み  
- **05_profile**：文明メンバーの文化的プロフィール  

これらはすべて **文明カテゴリ（CategoryChannel）単位** で動作する。

### 文明BOTの構造（レイヤー）

| レイヤー | 役割 |
|---------|------|
| **Features** | 文明の棚。各機能の目的・UI・動作を定義する中心。 |
| **DB Schema** | 文明の状態を永続化する。 |
| **State Machine** | 文明の状態遷移を管理する。 |
| **Commands** | ユーザー操作の入口。 |
| **UI Spec** | Embed / ボタン / メッセージの統一デザイン。 |
| **Operation Flow** | 各 Feature の処理フローを定義する。 |

このうち **03_features** は文明BOTの中心であり、  
すべての仕様は Feature を単位として整理される。

---

## 3. 開発の基本原則（Development Principles）

文明BOTの開発は、以下の原則に従う。

### ● Feature は文明の「棚」である
- 1 Feature = 1 文明の側面  
- Feature は独立して改善できる  
- Feature は UI / DB / Command / Flow を持つ

### ● 文明カテゴリ単位で動作する
- すべてのデータは CategoryChannel に紐づく  
- 文明外ではコマンドを実行できない  
- 文明は複数同時に存在できる

### ● UI は文明として統一された美しさを持つ
- Embed の色・構造・情報量は統一  
- 文明の状態は必ず Embed  
- 軽い通知はテキスト  
- UI の統一は文化の一部である

### ● コードは「文明の状態を正しく表現する」ことを最優先にする
- DB と UI は常に同期しているべき  
- 文明の状態遷移は StateMachine に従う  
- Feature の責務を越えた処理は行わない

### ● 仕様書修正はコードブロックで提示する（必須）
- 修正案は必ず ```md``` で出力  
- コピペで差し替え可能な形にする  
- 説明文は外に書いてもよいが、修正対象は必ずコードブロック内

---

## 4. 文明の UI 原則（UI Principles）

文明BOTの UI は、  
**視認性・統一性・文化的美しさ** を最重要とする。

### Embed を使用するケース（必須）
- 文明の状態を表す情報  
- クエスト作成・編集・完了  
- ランキング  
- 週次レポート  
- setup 完了  
- 重要イベント通知  
- エラーメッセージ（赤）

### テキストで十分なケース
- 軽い通知  
- register  
- admin の軽い設定変更  
- 内部的な成功通知

### 色の原則
- 成功：青  
- エラー：赤  
- 文明の状態：青 or 灰  
- 警告：黄色

## 5. カテゴリID取得の標準ルール（必須）

文明BOTは、すべての文明データを  
**Discord のカテゴリ（CategoryChannel）単位** で管理する。

### なぜ共通関数が必要か
Discord.js の型仕様では  
`interaction.channel` が複数の型を取り得るため、  
`parentId` を直接参照すると型エラーが発生する。

### 文明BOTの標準方針
すべてのコマンドは必ず  
**getCategoryId()** を使用してカテゴリIDを取得する。

### コマンド実装ルール
1. `getCategoryId(interaction.channel)` を呼ぶ  
2. null → 「文明カテゴリ内で実行してください」  
3. 取得した category_id を settings と照合する

## 6. Feature 開発の考え方（Feature Philosophy）

Feature は文明の「棚」であり、  
文明の側面を表す **文化的モジュール** である。

### Feature の構成要素
- 目的（Purpose）  
- UI（Embed / ボタン）  
- DB（テーブル構造）  
- Commands（操作）  
- Operation Flow（処理フロー）  
- Known Issues（既知の懸念点）

### Feature の改善方針
- Feature は単独で改善できる  
- 依存関係は overview が管理する  
- UI と DB の整合性を最優先  
- 文明の文化を壊さない変更を行う

---

## 7. 今後の拡張方針（Future Vision）

文明BOTは、以下の方向に拡張される。

- 文明の歴史アーカイブ  
- Web ダッシュボード  
- 文明間の比較  
- 自動週次レポート  
- 活動ログの可視化  
- 文明の成長曲線の分析  
- プロフィールの文化的深化  

文明BOTは、  
**コミュニティの文化を育てるための OS** として進化し続ける。
